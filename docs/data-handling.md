# Data Handling & PII

Prod-readiness **item 3**. The app records **named** individuals (role-holders and
survey respondents), so this documents what personal data is stored, where it
lives, how long it stays, how to erase it, and what leaves the box.

> **License / scope.** One private instance for the license holder. This is a
> personal professional workspace, not a multi-tenant service. The named data
> below is the practitioner's own change-management roster — treat it with the
> same care you'd give any HR-adjacent working notes.

## What personal data is stored

| Data | Table.column | Origin |
|------|--------------|--------|
| Role-holder name | `roles.person_name` | Entered by the practitioner on the roster |
| Survey respondent name | `survey_recipients.person_name` | **Snapshotted** from the roster at invite time |
| A respondent's answers | `survey_responses` (keyed by `recipient_id`) | Submitted by the respondent via their token link |
| Access tokens | `survey_recipients.token`, `projects.share_token` | Generated by the app; credentials, not PII, but treated as secret |

No email addresses, no free-text about third parties beyond the name, no
authentication accounts for respondents (they're reached only by opaque token).
Derived scores are **never stored** — always recomputed — so there is no
shadow copy of a person's results to clean up.

## Where it lives

- A single SQLite file on the Fly volume (`/data/proxima.db`), plus verified
  startup snapshots under `/data/backups` (pruned to the last 10). **Backups
  contain the same PII** — an erasure isn't fully complete until older snapshots
  age out or are cleared.
- Nothing is sent to any third party. The app makes no outbound calls; the CSP
  (`connect-src 'self'`) and `Referrer-Policy: no-referrer` keep tokens and data
  from leaking outward.

## Encryption at rest

**Decision.** The live database and its on-machine `/data/backups` are protected
by **Fly's volume encryption at rest** (volumes are encrypted by default). We do
**not** add application-level DB encryption (e.g. SQLCipher): for a single-user
instance the key would have to live on the same machine as the data (a Fly
secret / env var), so it adds little over volume encryption while bringing a
native-build burden and backup/restore complexity — YAGNI for this threat model.

The one place data leaves the encrypted volume is the **weekly off-Fly backup**
(`.github/workflows/backup.yml`), which lands as a GitHub artifact. That copy
**is** encrypted — AES-256 via `openssl enc` before upload, keyed by the
`BACKUP_ENCRYPTION_KEY` GitHub secret (held separately from the data). The
workflow refuses to upload if that secret is missing, so a plaintext database
can never leave. To restore from it:

```
openssl enc -d -aes-256-cbc -pbkdf2 -in proxima-backup.db.enc \
  -out proxima-backup.db -pass env:BACKUP_ENCRYPTION_KEY
```

then point the server at the decrypted file (or drop it in as `/data/proxima.db`
and reboot — the app boots off a backup file; see `infra/backup.ts`).

> Set the secret once: `gh secret set BACKUP_ENCRYPTION_KEY --body "$(openssl rand -hex 32)"`
> and keep that value somewhere safe — losing it makes existing off-Fly
> artifacts unrecoverable.

## Retention & erasure

Data is kept until explicitly removed. There is no automatic expiry of stored
answers (only the *link* tokens expire — see migration `012_token_expiry.sql`).
Erasure paths, narrowest first:

| To remove | Do this | Effect |
|-----------|---------|--------|
| One respondent (data-subject erasure) | `DELETE /api/survey-recipients/:id` | Removes that person's snapshot name + responses; the campaign and everyone else are untouched; the roll-up recomputes over the remaining respondents |
| One role-holder from the roster | delete the role | Clears `roles.person_name`; a **submitted** survey keeps its snapshot (by design — `role_id` is `SET NULL`), so also erase the recipient if full erasure is intended |
| A whole survey round | `DELETE /api/surveys/:id` | Cascades all recipients + responses for that campaign |
| Everything for a project | delete the project | Cascades all of the above |

After any erasure, PII may still persist in `/data/backups` until those
snapshots rotate out. For a hard erasure, also clear the relevant backups.

## What leaves the box (exports)

- **JSON export / project duplicate** (`GET /api/projects/:id/export`,
  `POST /api/import`) carries the **roster** (`roles`, including
  `person_name`) so a transferred project is complete. It does **not** include
  survey campaigns, recipients, or responses — survey PII never leaves via
  export. `share_token` is stripped on export (a credential, and it would
  collide on re-import).
- **CSV exports** are per-grid; the roles grid includes names by design.
- Exports are files the practitioner saves locally — treat them as containing
  names and store them accordingly.

## Practical guidance

- If a respondent asks to be removed, use `DELETE /api/survey-recipients/:id`
  and, if they were also on the roster and should be fully gone, delete the role.
- Keep exported JSON/CSV out of shared drives; they carry roster names.
- Rotating `CMT_SESSION_SECRET` logs everyone out but doesn't touch stored data.
